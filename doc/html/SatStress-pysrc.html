 14  1 Installation 
 15  ============== 
 16  Hopefully getting C{SatStress} to work on your system is a relatively painless 
 17  process, however, the software does assume you have basic experience with the 
 18  Unix shell and programming within a Unix environment (though it should work on 
 19  Windows too).  In particular, this installation information assumes you already 
 20  have and are able to use: 
 21   
 22    - compilers for both C and Fortran.  Development has been done on Mac OS X 
 23      (10.5) using the GNU compilers C{gcc} and C{g77}, so those should 
 24      definitely work.  On other systems, with other compilers, your mileage may 
 25      vary. 
 26   
 27    - the C{make} utility, which manages dependencies between files. 
 28   
 29  1.1 Other Required and Recommended Software 
 30  ------------------------------------------- 
 31  To get the L{SatStress} package working, you'll need to install some other 
 32  (free) software first: 
 33   
 34    - B{Python 2.5} or later (U{http://www.python.org}).  If you're running a 
 35      recent install of Linux, or Apple's Leopard operating system (OS X 10.5.x), 
 36      you already have this.  Python is also available for Microsoft Windows, and 
 37      just about any other platform you can think of. 
 38   
 39    - B{SciPy} (U{http://www.scipy.org}), a collection of scientific libraries 
 40      that extend the capabilities of the Python language. 
 41   
 42  In addition, if you want to use L{GridCalc}, you'll need: 
 43   
 44    - B{netCDF} (U{http://www.unidata.ucar.edu/software/netcdf/}, a library of 
 45      routines for storing, retrieving, and annotating regularly gridded 
 46      multi-dimensional datasets.  Developed by U{Unidata 
 47      <http://www.unidata.ucar.edu>} 
 48   
 49    - B{netcdf4-python} (U{http://code.google.com/p/netcdf4-python/}, a Python 
 50      interface to the netCDF library. 
 51   
 52  If you want to actually view L{GridCalc} output, you'll need a netCDF file 
 53  viewing program.  Many commercial software packages can read netCDF files, such 
 54  as ESRI ArcGIS and Matlab (from the Mathworks).  A simple and free reader for 
 55  OS X is U{Panoply <http://www.giss.nasa.gov/tools/panoply/>}, from NASA.  If 
 56  you want to really be able to interact with the outputs from this model, you 
 57  should install: 
 58   
 59    - B{Matplotlib/Pylab} (U{http://matplotlib.sourceforge.net/}), a Matlab-like 
 60      interactive plotting and analysis package, which uses Python as its 
 61      "shell".

 63  1.2 Building and Installing SatStress 
 64  ------------------------------------- 
 65   
 66  B{RESUME HERE} 
 67   
 68  Once you have the required software prerequisites installed, you should be able 
 69  to C{cd} into the directory containing the L{SatStress} module, and simply type 
 70  C{make all} at the command line.  This will compile the Love number code and 
 71  run the small L{SatStress.test} program embedded within L{SatStress}, just to 
 72  make sure that everything is in working order (or not).  If you're not using 
 73  the GNU Fortran 77 compiler C{g77}, you'll need to edit the C{Makefile} for the 
 74  Love number code:: 
 75   
 76    ./Love/JohnWahr/Makefile

 79  2 Design Overview 
 80  ================= 
 81  A few notes on the general architecture of the C{SatStress} package. 
 82   
 83  2.1 A Toolkit, not a Program 
 84  ---------------------------- 
 85    The C{SatStress} package is not itself a stand-alone program (or not much of 
 86    one anyway).  Instead it is a set of tools with which you can build programs 
 87    that need to know about the stresses on the surface of a satellite, and how 
 88    they compare to tectonic features, so you can do your own hypothesizing and 
 89    testing. 
 90   
 91  2.2 Object Oriented 
 92  ------------------- 
 93    The package attempts to make use of U{object oriented programming 
 94    <http://en.wikipedia.org/wiki/Object-oriented_programming>} (OOP) in order to 
 95    maximize the re-usability and extensibility of the code.  Many scientists are 
 96    more familiar with the U{imperative programming style 
 97    <http://en.wikipedia.org/wiki/Imperative_programming>} of languages like 
 98    Fortran and C, but as more data analysis and hypothesis testing takes place 
 99    inside computers, and as many scientists become highly specialized and 
100    knowledgeable software engineers (even if they don't want to admit it), the 
101    advantages of OOP become significant.  If the object orientation of this 
102    module seems odd at first glance, don't despair, it's worth learning. 
103   
104  2.3 Written in Python 
105  --------------------- 
106    U{Python <http://www.python.org>} is a general purpose, high-level scripting 
107    language.  It is an interpreted language (as opposed to compiled languages 
108    like Fortran or C) and so Python code is very portable, meaning it is usable 
109    on a wide variety of computing platforms without any alteration.  It is 
110    relatively easy to learn and easy to read, and it has a very active 
111    development community.  It also has a large base of friendly, helpful 
112    scientific users and an enormous selection of pre-existing libraries designed 
113    for scientific applications.  For those tasks which are particularly 
114    computationally intensive, Python allows you to extend the language with code 
115    written in C and Fortran.  Python is also U{Free Software 
116    <http://www.gnu.org/philosophy/free-sw.html>}.  If you are a scientist and 
117    you write code, Python is a great choice. 
118   
119  2.4 Open Source 
120  --------------- 
121    Because science today is intimately intertwined with computation, it is 
122    important for researchers to share the code that their scientific results are 
123    based on.  No matter how elegant and accurate your derivation is, if your 
124    implementation of the model in code is wrong, your results will be flawed. 
125    As our models and hypotheses become more complex, our code becomes vital 
126    primary source material, and it needs to be open to peer review.  Opening our 
127    source: 
128   
129      - allows bugs to be found and fixed more quickly 
130      - facilitates collaboration and interoperability 
131      - reduces duplicated effort 
132      - enhances institutional memory 
133      - encourages better software design and documentation 
134   
135    Of course, it also means that other people can use our code to write their 
136    own scientific papers, but I{that is the fundamental nature of science}.  We 
137    are all "standing on the shoulders of giants".  Nobody re-derives quantum 
138    mechanics when they just want to do a little spectroscopy.  Why should we all 
139    be re-writing each others code I{ad nauseam}?  Opening scientific source code 
140    will ultimately increase everyone's productivity.  Additionally, a great deal 
141    of science is funded by the public, and our code is a major product of that 
142    funding.  It is unethical to make it proprietary.
